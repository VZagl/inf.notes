## Общая информация

**Версия:** v0.8

## Определение

Memory Bank System — это модульная, ориентированная на документацию система управления задачами, использующая пользовательские режимы Cursor 2.0 для обеспечения постоянной памяти и структурированного рабочего процесса разработки с визуальными картами процессов.

## Основные компоненты

### 1. Команды Cursor 2.0

Система использует набор специализированных команд для управления различными фазами разработки:

- **`/van`** — Инициализация проекта

  - Начальная настройка проекта
  - Подготовка структуры Memory Bank

- **`/plan`** — Планирование задач

  - Создание и структурирование задач
  - Определение приоритетов и зависимостей

- **`/creative`** — Дизайн и креативные решения

  - Генерация идей и концепций
  - Проектирование архитектуры

- **`/build`** — Реализация и разработка

  - Непосредственная разработка кода
  - Внедрение функциональности

- **`/reflect`** — Обзор и анализ

  - Анализ выполненной работы
  - Выявление проблем и улучшений

- **`/archive`** — Документирование и архивирование
  - Сохранение завершенных задач
  - Подготовка документации

### 2. Иерархическая загрузка правил

**Принцип работы:**

- Система загружает только те правила, которые необходимы для текущего этапа разработки
- Правила загружаются динамически в зависимости от активной команды
- Обеспечивает адаптивность и эффективность работы

**Преимущества:**

- Оптимизация использования токенов
- Снижение когнитивной нагрузки на AI
- Более точное и релевантное руководство на каждом этапе

### 3. Визуальные карты процессов

- Графическое представление рабочих процессов
- Наглядное отображение взаимосвязей между этапами
- Упрощение навигации и понимания структуры проекта

### 4. Оптимизация токенов

- Эффективное использование токенов для повышения производительности
- Снижение затрат на обработку
- Загрузка только необходимых правил и контекста

## Процесс работы с системой

### Общий workflow

Система работает по циклическому процессу, где каждая команда выполняет свою роль в общей структуре разработки:

```
/van → /plan → /creative → /build → /reflect → /archive
  ↑                                                      ↓
  └──────────────────────────────────────────────────────┘
```

### Детальное описание этапов

#### 1. Инициализация проекта (`/van`)

**Цель:** Подготовка проекта к работе

**Действия:**

- Определение платформы проекта (веб, мобильное приложение, десктоп и т.д.)
- Оценка сложности задач
- Создание базовой структуры Memory Bank
- Инициализация файлов контекста

**Результат:** Проект готов к планированию, создана структура для хранения информации

#### 2. Планирование (`/plan`)

**Цель:** Структурирование задач и создание плана реализации

**Действия:**

- Анализ требований проекта
- Разбиение на задачи и подзадачи
- Определение приоритетов и зависимостей
- Создание детальных планов реализации на основе уровня сложности
- Заполнение `tasks.md` и `progress.md`

**Результат:** Четкий план работы с приоритизированными задачами

#### 3. Творческий этап (`/creative`)

**Цель:** Проектирование решений для сложных компонентов

**Действия:**

- Исследование вариантов дизайна
- Проектирование архитектуры компонентов
- Генерация идей и концепций
- Создание эскизов и прототипов
- Сохранение решений в директории `creative/`

**Результат:** Проектные решения готовы к реализации

#### 4. Реализация (`/build`)

**Цель:** Непосредственная разработка кода

**Действия:**

- Систематическое выполнение запланированных изменений
- Написание кода согласно плану
- Внедрение функциональности
- Обновление `activeContext.md` с текущим фокусом работы
- Регулярное обновление `progress.md`

**Результат:** Реализованная функциональность

#### 5. Рефлексия (`/reflect`)

**Цель:** Анализ выполненной работы и извлечение уроков

**Действия:**

- Анализ выполненной работы
- Выявление проблем и узких мест
- Документирование извлеченных уроков
- Оценка качества реализации
- Обновление `systemPatterns.md` с новыми паттернами
- Подготовка рекомендаций для будущих итераций

**Результат:** Проанализированная работа с выводами и улучшениями

#### 6. Архивация (`/archive`)

**Цель:** Документирование и сохранение завершенных задач

**Действия:**

- Создание полной документации проекта
- Перемещение завершенных задач в `archive/`
- Обновление Memory Bank с финальной информацией
- Формирование итогового отчета
- Очистка активных файлов для новых задач

**Результат:** Завершенный проект с полной документацией

### Взаимодействие с Memory Bank

**Ключевой принцип:** Все команды взаимодействуют с общей директорией `memory-bank/`, обеспечивая сохранение контекста на протяжении всего рабочего процесса.

**Механизм работы:**

1. При активации команды загружаются только релевантные правила для текущей фазы
2. Каждая команда читает и обновляет соответствующие файлы Memory Bank
3. Контекст сохраняется между сессиями через файлы Memory Bank
4. AI использует информацию из Memory Bank для понимания текущего состояния проекта

### Цикличность процесса

Процесс может быть:

- **Линейным:** Последовательное прохождение всех этапов от `/van` до `/archive`
- **Итеративным:** Возврат к предыдущим этапам для доработки (например, `/build` → `/reflect` → `/plan` → `/build`)
- **Селективным:** Использование только необходимых команд в зависимости от текущих потребностей

### Преимущества такого подхода

- **Сохранение контекста:** Информация не теряется между сессиями работы
- **Структурированность:** Четкое разделение фаз помогает организовать работу
- **Эффективность:** Загружаются только необходимые правила, оптимизируя использование токенов
- **Гибкость:** Возможность адаптации процесса под конкретный проект

## Структура файлов Memory Bank

Система использует директорию `memory-bank/` для хранения всей информации о проекте. Все команды взаимодействуют с этой директорией, обеспечивая сохранение контекста.

### Основные файлы

- **`projectbrief.md`** — Основной документ проекта

  - Формирует основу для всех остальных файлов
  - Содержит общее описание проекта
  - Определяет цели и границы проекта

- **`productContext.md`** — Контекст продукта

  - Объясняет цель проекта
  - Описывает решаемые проблемы
  - Определяет целевую аудиторию и требования

- **`activeContext.md`** — Активный контекст работы

  - Отражает текущий фокус работы
  - Последние изменения и их влияние
  - Следующие шаги и приоритеты
  - Активная фаза разработки

- **`systemPatterns.md`** — Архитектурные паттерны

  - Описывает архитектуру системы
  - Технические решения и обоснования
  - Шаблоны проектирования
  - Принятые соглашения

- **`techContext.md`** — Технологический контекст

  - Используемые технологии и их версии
  - Среда разработки и инструменты
  - Ограничения и требования
  - Конфигурации и настройки

- **`progress.md`** — Отслеживание прогресса

  - Выполненные задачи
  - Оставшаяся работа
  - История изменений
  - Метрики выполнения
  - Статистика проекта

- **`tasks.md`** — Список задач (альтернативный формат)

  - Текущие задачи и их статусы
  - Приоритеты и зависимости
  - Назначенные ответственные

- **`memory_bank_instructions.md`** — Инструкции для AI-агента
  - Правила работы с Memory Bank
  - Конвенции именования
  - Форматы данных
  - Рекомендации по обновлению файлов

### Директории

- **`creative/`** — Креативные материалы

  - Идеи и концепции
  - Эскизы и прототипы
  - Дизайн-решения
  - Альтернативные варианты

- **`archive/`** — Архив завершенных задач
  - Завершенные задачи и их документация
  - Исторические данные проекта
  - Документация по завершенным проектам
  - Извлеченные уроки

### Взаимосвязь файлов

```
projectbrief.md (основа)
    ↓
productContext.md + techContext.md
    ↓
activeContext.md (обновляется постоянно)
    ↓
systemPatterns.md (формируется в процессе)
    ↓
progress.md (отслеживает изменения)
    ↓
archive/ (финальное сохранение)
```

## Особенности системы

### Адаптивность

- Система разработана для адаптации AI под конкретные нужды пользователя
- Возможность модификации правил через Cursor AI
- Гибкая настройка под индивидуальный рабочий процесс

### Постоянная память

- Сохранение контекста между сессиями
- Отслеживание прогресса проекта
- Преемственность работы над задачами

### Структурированный рабочий процесс

- Четкое разделение фаз разработки
- Последовательное выполнение задач
- Контроль качества на каждом этапе

## Примечания разработчика

Memory Bank — это личный хобби-проект, разработанный для собственного использования в кодинге.

**Особенности проекта:**

- Не ведется трекер проблем (issues)
- Не собирается активная обратная связь
- Основное преимущество — возможность попросить Cursor AI напрямую изменить или обновить правила под конкретный рабочий процесс

**Философия:**
Система разработана так, чтобы быть адаптируемой AI, позволяя пользователям настраивать её под свои нужды без необходимости внешней поддержки.

Система предназначена для:

- Структурирования процесса разработки
- Управления задачами в проектах
- Обеспечения постоянной памяти между сессиями работы
- Оптимизации взаимодействия с Cursor AI
