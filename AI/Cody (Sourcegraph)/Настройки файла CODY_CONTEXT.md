# Настройка файла CODY_CONTEXT.md для нового проекта

## Что такое CODY_CONTEXT.md?

Файл `CODY_CONTEXT.md` - это специальный конфигурационный файл, который определяет правила работы AI-ассистента Cody в вашем проекте. Это как "инструкция по эксплуатации" для ИИ, которая помогает ему понимать специфику вашего проекта и отвечать именно так, как вам нужно.

## Зачем нужен этот файл?

По умолчанию Cody работает в общем режиме, но каждый проект уникален:

- Разные языки программирования
- Различные стили кодирования
- Специфические требования к документации
- Корпоративные стандарты
- Предпочтения команды разработчиков

## Пошаговая настройка файла

### Шаг 1: Создание файла

```bash
touch CODY_CONTEXT.md
```

````
### Шаг 3: Настройка языка интерфейса

Определите, на каком языке должен отвечать Cody:

```markdown
### Язык общения:

- **Основной язык ответов**: русский/английский
- **Комментарии в коде**: русский/английский
- **Документация**: русский/английский
- **Commit сообщения**: русский/английский
````

### Шаг 4: Стандарты кодирования

Укажите правила для вашего проекта:

```markdown
### Стиль кода:

- **Отступы**: 2/4 пробела или табы
- **Кавычки**: одинарные/двойные
- **Точки с запятой**: обязательны/опциональны
- **Именование**: camelCase/snake_case/kebab-case
- **Максимальная длина строки**: 80/100/120 символов
```


## Частые ошибки при настройке

### ❌ Неправильное расположение файла

Файл должен быть в корне проекта, не в подпапке

### ❌ Неправильный формат Markdown

Проверьте синтаксис заголовков и списков

### ❌ Слишком строгие ограничения

Не отключайте все функции сразу, тестируйте постепенно

### ❌ Противоречивые настройки

Убедитесь, что правила не конфликтуют между собой

## Дополнительные советы

### Итеративный подход

Не пытайтесь настроить все сразу. Начните с базовых настроек и постепенно добавляйте специфичные для проекта правила.

### Командная работа

Если работаете в команде, обсудите настройки с коллегами и зафиксируйте их в файле для единообразия.

### Документирование изменений

Ведите историю изменений в самом файле, чтобы понимать, почему были приняты те или иные решения.

### Регулярное обновление

Периодически пересматривайте настройки по мере развития проекта и изменения требований.
